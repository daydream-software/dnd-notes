## Detailed Gates

### TIER 1: RESTORE SAFETY (#40) ŌĆö Blocks Everything Else

**Context:** Currently, `POST /api/admin/restore` swaps the SQLite database immediately. Connected clients don't know the restore happened and continue with stale state.

**What Must Work:**
1. **Active writes during restore are blocked or queued**
   - Test: Concurrent `POST /api/notes` requests, trigger restore, verify writes return 503 or 429 (not silently lost)
   - Risk: If skipped, users lose work without notification

2. **Active reads during restore return 503 or redirect**
   - Test: `GET /api/campaigns` in-flight, restore happens, verify 503 response (not hanging)
   - Risk: If skipped, clients hang or receive stale data

3. **Web UI detects maintenance mode and shows notification**
   - Test: UI polls `/ready` every 5s, receives 503, displays "Tenant is updating" banner
   - Risk: If skipped, UI appears broken; users don't know maintenance is in progress

4. **Restore state is transitioned + logged (control-plane audit trail)**
   - Test: Control plane moves tenant from `ready` ŌåÆ `restoring` ŌåÆ `ready`, all transitions logged
   - Risk: If skipped, no audit record of what happened during downtime

5. **Connection timeouts don't block restore completion**
   - Test: Simulate long-running restore (>30s), verify statement timeout extends or restore uses separate pool
   - Risk: If skipped, restore can hang if any query takes >30s

**Ship Criteria:** `npm run test` passes all 5 restore gates + existing health/readiness suite still green.

### TIER 2: PER-TENANT POSTGRES ROLES (#69) ŌĆö After #40

**Context:** Currently, all tenants share one Postgres role. If any tenant app is compromised, attacker reads all tenants' notes.

**What Must Work:**

1. **Per-tenant role is created during provisioning with randomized password**
   - Test: `provisioning.test.ts` new case: `ensureTenantDatabase()` returns `{ databaseName, perTenantSecret }` with role name, username, password
   - Acceptance: Control-plane test double verifies secrets are generated per tenant
   - Risk: If skipped, credentials remain shared; multi-tenant isolation is theater

2. **Per-tenant role has minimal privileges (no CREATE/DROP/ALTER databases)**
   - Test: New isolation test: tenant role tries `CREATE DATABASE`, `DROP DATABASE`, `ALTER ROLE`ŌĆöall fail with permission denied
   - Acceptance: Each attempt logged; only CONNECT + USAGE on public schema granted
   - Risk: If skipped, compromised tenant app can destroy other tenants' databases

3. **Per-tenant credentials are mounted as per-tenant Kubernetes secrets**
   - Test: Deployment manifest generated by `buildTenantInfrastructureBundle()` has env ref to `tenant-{id}-postgres-credentials`
   - Acceptance: Fake K8s infrastructure manager confirms secret name isolation
   - Risk: If skipped, secrets shared across pods; credential rotation cascades to all tenants

4. **Cross-tenant isolation verified: tenant A role cannot read tenant B schema**
   - Test: Provision two tenants, connect as tenant A role, try `SELECT * FROM public.notes` in tenant B databaseŌĆöfails with permission denied
   - Acceptance: Query logs show access denied; no leakage
   - Risk: If skipped, SQL injection in tenant A app could read tenant B notes

5. **Deprovisioning drops per-tenant role (after terminating active connections)**
   - Test: New deprovision test: `deleteTenantDatabase()` terminates sessions on role, then `DROP ROLE tenant_abc123`
   - Acceptance: Post-deprovision, old role doesn't exist
   - Risk: If skipped, orphaned roles accumulate; Postgres cleanup overhead grows

6. **Credential rotation invalidates old password + logs audit event**
   - Test: New rotation test: trigger credential rotation, old password fails on new connections, new password works, control-plane audit log records `{ role, event: "rotated", timestamp, initiator }`
   - Acceptance: Audit entry present; old token rejected within 1s
   - Risk: If skipped, stale credentials in leaked secrets remain valid indefinitely

7. **Migration path documented + tested: existing shared credentials ŌåÆ per-tenant**
   - Test: New migration test: start with shared `TENANT_DATABASE_RUNTIME_URL`, run migration script, verify all existing tenants now have per-tenant secrets
   - Acceptance: Downtime measured; old shared creds deactivated
   - Risk: If skipped, upgrade requires manual per-tenant secret creation; error-prone

**Ship Criteria:** `npm run test` passes all 7 gates + existing provisioning suite still green + no test flakiness in credential rotation.

### TIER 3: KEYCLOAK OIDC (#56) ŌĆö After #40, Parallel with #69

**Context:** Currently, auth is app-scoped (bearer tokens in `Authorization` header). Need centralized Keycloak for workforce identity.

**What Must Work:**

1. **Keycloak realm model: admin realm, customer realm, per-control-plane client**
   - Test: Integration test with FakeKeycloakServer: admin realm issues control-plane OIDC client, customer realm issues tenant OIDC client, realm isolation verified
   - Acceptance: Token from admin realm rejected by tenant app, vice versa
   - Risk: If skipped, token scope bleed allows tenant app to read admin credentials

2. **Control-plane auth flow: OIDC redirect ŌåÆ token ŌåÆ /internal routes authenticated**
   - Test: CP calls `/internal/tenants`, redirected to Keycloak, receives token, re-authenticates, succeeds
   - Acceptance: `/internal/tenants` returns 200 with valid bearer token; 401 without
   - Risk: If skipped, /internal routes open to Internet; security boundary lost

3. **Tenant auth flow: OIDC redirect ŌåÆ token bound to tenant context**
   - Test: User visits `tenant-abc123.dnd-notes.app`, redirected to Keycloak, token issued, authenticates `/api/campaigns` *on that tenant*
   - Acceptance: Token issued with tenant claim; campaigns list reflects login
   - Risk: If skipped, users can't log in to tenants; feature incomplete

4. **Token validation: expired/malformed/missing tokens consistently return 401**
   - Test: API middleware validates `Authorization: Bearer {token}` signature, expiry, issuer; missing/invalid/expired all return 401 with consistent error message
   - Acceptance: No 500 errors from token parsing; all 401s logged
   - Risk: If skipped, malicious tokens could trigger app crashes

5. **No auth token leakage in logs or metrics**
   - Test: Grep control-plane logs, tenant app logs, metrics endpoints for token fragments (first 20 chars of JWT); zero findings
   - Acceptance: Auth logs redact tokens; metrics don't capture bearer headers
   - Risk: If skipped, leaked logs in bug reports expose live tokens to forum readers

6. **Tenant authorization remains local (OIDC provides identity, app enforces scope)**
   - Test: User from admin realm tries to access tenant app with admin tokenŌĆörejected; token is only identity proof, scope/membership enforced by app
   - Acceptance: 403 Forbidden (authorization) not 401 Unauthorized (identity)
   - Risk: If skipped, Keycloak group membership bleeding into tenant scope

7. **Guest token coexistence: X-Guest-Token still works alongside OIDC**
   - Test: Two flows: (1) Authenticated user with Bearer token, (2) Guest on shared link with X-Guest-Token; both work, no cross-pollution
   - Acceptance: Middleware handles both headers; no regression in guest flows
   - Risk: If skipped, existing share-link workflows break; backward compatibility lost

**Ship Criteria:** `npm run test` passes all 7 gates + existing auth/security suite still green + integration test with real Keycloak container passes.

## Highest-Risk Failures If Team Starts with #69 First

### Failure Mode 1: Silent Restore with Stale Auth
User editing note ŌåÆ restore rotates per-tenant credentials ŌåÆ tenant pod hasn't re-read secret ŌåÆ next API call fails with 401 instead of "Maintenance in progress" ŌåÆ User thinks account is locked.

### Failure Mode 2: Orphaned Postgres Connections
Credential rotation happens mid-flight ŌåÆ connection pool holds old credentials ŌåÆ new requests use old password ŌåÆ pool saturated with dead connections ŌåÆ app appears hung.

### Failure Mode 3: OIDC Realm Token Confusion
OIDC admin token leaked in shared environment ŌåÆ second tenant's pod reads it ŌåÆ second tenant can auth as admin to Postgres (while shared creds still in place) ŌåÆ tenant B reads tenant A notes.

### Failure Mode 4: Restore Credential Sync Race
Per-tenant credential rotation initiated ŌåÆ Kubernetes secret updated ŌåÆ Pod drained (SIGTERM) ŌåÆ in-flight queries use stale credential in process memory ŌåÆ queries fail during shutdown ŌåÆ pod hangs 30s.

**All four prevented by shipping #40 first.**
