## Mikey (Lead) — Architecture Direction

**Role:** Lead  
**Input:** Issue #42, existing repo decisions, FFMikha's expanded vision for SaaS multi-instance  
**Outcome:** Recommended #42 as a thin provisioning/isolation spike, deferring platform layers

### Key Recommendations
- **Scope IN for #42:** Container lifecycle, SQLite volumes, Docker Compose, routing stub, control-plane contract
- **Scope OUT for #42:** Keycloak, Kubernetes operator, portal app, billing, wildcard ingress
- **Build order:** Prove instance lifecycle and cost before adding SSO, operator, or portal
- **Architecture:** Multi-instance SaaS with Keycloak + portal is credible target; #42 validates foundation only

### Decision Points
1. **Container per customer** (recommended over process-per-host) — strongest isolation, K8s-ready
2. **Provisioning script as production code** — not throwaway
3. **OIDC as alternative auth** — add alongside localStorage first via feature flag
4. **Subdomain routing** (`alice.dndnotes.app`) — full origin isolation, avoids path-prefix bugs
5. **Minimal control plane** — registry, provisioning, instance lifecycle only

### Deliverable Shape
- `provisioning/` directory with shell script or Node CLI for create/start/stop/destroy
- Parameterized Docker Compose template per instance
- README with measured results and go/no-go findings

## Brand (Platform Dev) — Infrastructure Evaluation

**Role:** Platform Dev  
**Input:** Issue #42, deployment/ops decisions, repo same-origin and backup/restore preferences  
**Outcome:** Recommended simple app-per-customer provisioning; defer Keycloak and operator

### Control Plane Ownership
- Customer signup/subscription state
- Instance create, suspend, delete, bootstrap
- Domain/subdomain assignment and routing metadata
- Health/status, version tracking, upgrade orchestration
- Backup inventory / restore policy
- Global identity only if shared login becomes real need

### Tenant Instance Ownership
- Campaign, note, membership, share-link data
- Per-instance auth/session handling (until shared SSO justified)
- Backup/restore execution against instance's SQLite
- Maintenance/read-only behavior during restore/upgrade
- Product-specific admin actions local to tenant

### Keycloak Assessment
**Good fit when:** same human needs multi-instance access, enterprise SSO required, centralized user lifecycle critical  
**Operational costs:** another stateful service, client/realm automation per instance, single-service login blast radius  
**Recommendation:** Do NOT include in prototype; revisit only after instance lifecycle is proven

### Kubernetes Operator Assessment
**Worth it when:** instance count large, provisioning happens frequently, automated DNS/TLS/secrets/storage/upgrades needed  
**Overkill when:** small customer count, validating isolation model, one person can still support scripted provisioning  
**Recommendation:** Use plain provisioning automation first; move to K8s only when ops pain is concrete

### Smallest Meaningful Prototype
1. Create new customer instance from template
2. Assign public URL / same-origin config
3. Attach dedicated SQLite storage
4. Run health checks and expose instance status
5. Exercise backup and restore on one instance
6. Measure cold start, ready time, backup, restore, concurrent usage
7. Prove one upgrade rollout across multiple isolated instances

## Data (Backend Dev) — Backend Architecture & Boundary

**Role:** Backend Dev  
**Input:** Issue #42, backend architecture, auth/storage decisions  
**Outcome:** Recommended strict control-plane vs tenant-instance boundary with local per-instance authorization

### Control Plane Responsibilities
- Account, organization, subscription records
- Tenant/instance registry (tenantId, instanceId, status, version, domain, backup target)
- Provisioning and deprovisioning workflows
- Ingress/domain/TLS wiring
- Upgrade rollout and version tracking
- Fleet-level health and telemetry
- Support/admin access policy

### Tenant Instance Responsibilities
- Existing app API and SQLite data model
- Local user projection for authenticated users (keyed by IdP subject)
- Campaign-level membership and roles
- Share-link and guest-token flows
- Local backup/restore execution
- Maintenance/read-only behavior during restore/upgrade

### Control Plane → Instance Contract
- `GET /internal/status` — health, version, db mode, backup freshness
- `POST /internal/bootstrap` — one-time init with tenant metadata and initial admin subject
- `POST /internal/maintenance` — enter/leave read-only mode
- `POST /internal/backup` / `POST /internal/restore` — or worker-triggered hooks
- `POST /internal/reconcile-identity` — first-login/admin-seeding edge cases

### Identity Model
- **Shared IdP** (Keycloak) handles real-user login
- **Portal and tenant instances** are separate OIDC clients
- **Stable identity key:** issuer + subject (not email alone)
- **Global roles** live in control plane (support admin, billing admin, platform ops)
- **Local roles** live in tenant instance (site admin, campaign owner, guest)
- **Guest/share-link flows** remain instance-local, bypass SSO
- **Claiming guest membership** binds local membership row to authenticated SSO subject

### Current Pressure Points (Single-Tenant App)
1. One `NoteStore`, one SQLite file, all data in one DB
2. Email/password auth instead of OIDC
3. `SITE_ADMIN_EMAILS` env var too blunt for multi-tenant SaaS
4. `PUBLIC_WEB_URL`, `ALLOWED_ORIGINS` per-process; multi-customer needs flexibility
5. Default campaign assumption breaks in multi-tenant context
6. Restore swaps live DB in place, invalidates sessions (risky at fleet scale)

### Measurements Spike Must Capture
1. **Provisioning latency:** request accepted → healthy → bootstrap complete → usable
2. **Startup behavior:** cold start, warm restart, post-migration/restore
3. **SQLite mode:** WAL vs rollback journal, read/write concurrency under realistic traffic
4. **Backup/restore:** snapshot size, duration, operator steps, session fallout, required read-only window
5. **Upgrade fan-out:** migration per instance, failure handling, rollback, version-skew visibility
6. **Identity path timing:** portal login, redirect to tenant app, first-login linking, guest membership claim
7. **Per-instance cost:** idle memory/CPU, storage footprint, quiet instances per host
8. **Ingress/domain timing:** DNS/TLS readiness, failure modes for customer URLs
9. **Operational observability:** version, last backup, health, provisioning failures without manual instance access

### Minimum Provisioning Workflow for #42
1. **Control-plane request:** create tenant with slug/domain + initial admin email/subject
2. **Provisioning worker:** allocate config, create storage, deploy instance, apply env, wait for `/health`
3. **Bootstrap call:** record tenantId/instanceId, seed instance admin, mark complete
4. **Registry update:** `provisioning → ready` with version, endpoint, timestamps
5. **Portal handoff:** admin can launch their instance
