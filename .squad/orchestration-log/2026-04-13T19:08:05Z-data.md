## Inspection Summary

Data reviewed backend architecture in `apps/api/src/` (types.ts, validation.ts, note-store.ts) and cross-referenced web types. Prepared a comprehensive Phase 2a/2b/2c reference implementation plan.

### Key Findings

1. **Current State (Phase 1 Complete)**
   - `linkedNoteIds: string[]` stored in SQLite as `notes.linked_notes_json`
   - API exposes Note type with linkedNoteIds validation
   - `GET /api/notes/:noteId/backlinks` returns incoming references
   - Validation enforces 20-link limit, same-campaign-only, non-existent blocking

2. **Gaps for Phase 2**
   - No body-derived references (markdown inline syntax not yet in storage)
   - No dual-source reconciliation between linkedNoteIds and body markup
   - Frontend has no editor UI for adding references
   - No structured reference table for efficient querying and rename safety

3. **Recommendation: Staged Migration**

   **Phase 2a: First-class references table**
   - Add `note_references` table as single source of truth
   - `reference_type` column splits implicit (body-derived) vs. explicit (old linkedNoteIds)
   - Safe migration window: old and new systems coexist during transition
   - Lazy sync on first access; no immediate data conversion required

   **Phase 2b: Body parser integration**
   - Frontend (Lexical) parses markdown `[[note-id]]` syntax
   - On save: frontend sends both `body` and computed `inlineReferences` array
   - Backend validates inlineReferences, normalizes to references table
   - API contract extension: add optional `inlineReferences` field to NoteInput

   **Phase 2c: Query layer expansion**
   - New endpoints: `GET /api/notes/:noteId/references`, `GET /api/notes/:noteId/incoming-references`
   - Reference-aware search: filter notes by reference membership
   - No schema changes; builds on Phase 2a table

### Why This Approach Is Safe

1. **Zero breaking change** — existing `linkedNoteIds` API continues working
2. **Decouples concerns** — markdown stays plain text; references are structural
3. **Handles partial adoption** — until Lexical ships, references come from old field only
4. **Testable** — reference table is pure SQL; parser logic is unit-testable
5. **Search-friendly** — normalized structure enables efficient graph queries
