## Summary

End-to-end implementation and approval cycle for guest account claim flow on shared routes. Multiple revision cycles refined the solution to preserve membership identity, invalidate old guest tokens, and grant authenticated access to claimed campaigns.

## Timeline

### QA Gate (Chunk)
- Identified two critical acceptance criteria:
  1. Preserve membership identity as stable actor for note attribution after claim
  2. Invalidate old guest token to prevent anonymous backdoor post-claim
- Proposed regression coverage scenarios for gate enforcement

### First Pass (Brand)
- Implemented same-browser claim flow: `POST /api/shared/:shareToken/membership/claim`
- Linked real account by setting `campaign_memberships.user_id` on guest membership
- **Issue found:** Old guest token still authenticated shared routes after claim

### First Revision (Chunk → Data)
- Chunk rejected Brand's pass due to security issue (stale guest token)
- Data revised to rotate guest token on claim and replace token in browser

### Second Revision (Chunk → Stef)
- Chunk rejected Data's revision: linked account could not access claimed campaign through authenticated routes
- Stef revised implementation to:
  - Unlock `/api/campaigns`, `/api/campaigns/:campaignId`, `/api/overview`, `/api/notes` for linked memberships
  - Preserve owner-only restrictions on campaign settings and share-link management
  - Store claimed campaign as selected workspace in signed-in app
  - Keep note attribution on original guest membership
  - Invalidate pre-claim guest token

## Key Decisions Merged

1. **Guest token rotation on claim** — Old token invalidated, new token returned to browser for continued shared-route session
2. **Authenticated access for claimed memberships** — Claimed accounts gain read/write access to linked campaign
3. **Preserved membership identity** — Note attribution remains on original guest membership; collaborators cannot gain owner permissions
4. **Same-browser release scope** — No per-person links or cross-device flows in v1
