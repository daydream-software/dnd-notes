# 2026-04-17T23:52:56Z: Issue #42 Multi-Instance Design Spike

**Agents:** Mikey (Lead), Brand (Platform Dev), Data (Backend Dev)  
**Topic:** Architecture, infrastructure, and backend design for #42 multi-instance SaaS  
**Consensus:** Thin provisioning spike; defer Keycloak, operator, portal, billing

## Outcomes

### Mikey: Architecture direction
- **Scope #42:** Container lifecycle, SQLite volumes, Docker Compose, routing stub, control-plane contract
- **Defer:** Keycloak, operator, portal, billing, wildcard ingress
- **Build order:** Prove instance lifecycle before platform layers
- **Key calls:** Container-per-customer isolation, production-path provisioning script, subdomain routing, minimal control plane

### Brand: Infrastructure evaluation
- **Control plane:** Registry, customer lifecycle, instance provisioning, routing metadata, fleet health
- **Tenant instances:** One app per customer, local SQLite, independent operation
- **Keycloak:** Defer until enterprise SSO or multi-instance access justifies operational cost
- **Operator:** Defer until instance count/churn justifies K8s over scripted provisioning
- **Prototype shape:** Create, configure, health-check, backup/restore, measure cold-start/upgrade

### Data: Backend architecture
- **Boundary:** Control plane owns customer lifecycle; tenant owns app data + local auth
- **Identity:** Shared IdP for authentication, local authorization in each instance
- **Contract:** Narrow `/internal/` endpoints (status, bootstrap, maintenance, backup/restore, reconcile)
- **Metrics to capture:** Provisioning latency, startup, SQLite mode, backup/restore, upgrade fan-out, identity timings, cost envelope, observability
